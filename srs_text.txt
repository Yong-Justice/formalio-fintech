SECTION 3             —             COMPLETE FUNCTIONAL REQUIREMENTS                                  NOTE:            Every feature in this section is specified with: purpose, workflow, user interaction, backend behavior, APIs needed, edge cases, validations, dependencies, and technical considerations.                    3.1 Smart Bookkeeping Engine                    3.1.1 Income &amp; Expense Tracking                  Purpose: Enable SME owners to record all financial transactions quickly and accurately without accounting expertise.                  Workflow: User taps &quot;+&quot; button on home dashboard          →          selects Income or Expense          →          enters amount (FCFA)          →          selects or confirms AI-suggested category          →          adds optional description, photo, or receipt          →          saves. Transaction is immediately stored locally (SQLite via PowerSync) and synced to Supabase when online.                  Validations: Amount must be positive numeric value in FCFA. Category is required (AI suggests automatically). Description optional (max 500 characters). Date defaults to current timestamp but can be back-dated up to 90 days. Amount cap alert at 50,000,000 FCFA (suspicious transaction flag).                  Edge Cases: Duplicate transaction detection (same amount, same category, same date within 5 minutes triggers a warning). Network loss during save: transaction saved locally, queued for sync. Partial sync failure: retry with exponential backoff. Currency format: always stored as integer FCFA (no decimals          —          smallest unit is 1 FCFA).                    3.1.2 Voice Transaction Logging                  Purpose: Enable users to log transactions hands-free in French, English, or Cameroonian Pidgin          —          particularly useful for market traders and service providers who are actively working.                  Workflow: User taps microphone icon          →          system prompts &quot;Dites votre transaction&quot;          →          user speaks (e.g., &quot;J&apos;ai vendu 10 cartons de sardines pour 45,000 francs&quot;)          →          Whisper API transcribes          →          Claude Haiku categorizes          →          screen shows pre-filled transaction form          →          user confirms or edits          →          saves.                  APIs: OpenAI Whisper API (audio transcription), Claude Haiku API (categorization). Fallback: Whisper.cpp running locally for offline transcription of queued recordings.                  Technical Considerations: Audio recorded at 16kHz mono (optimal for Whisper). Max recording: 60 seconds. Audio file compressed to .webm before upload. Offline mode: audio saved locally in base64, transcription deferred until connectivity restored. Pidgin accuracy: ~85% with Whisper; system prompt includes Pidgin          →          French glossary for categorization improvement.                    3.1.3 AI Transaction Categorization                  Purpose: Automatically categorize transactions according to SYSCOHADA chart of accounts, eliminating the need for accounting knowledge.                  Workflow: Transaction description sent to Claude Haiku API with system prompt containing SYSCOHADA categories + Cameroon business context          →          model returns category ID + confidence score + SYSCOHADA code          →          if confidence &gt; 85%: auto-applied; if &lt; 85%: top 3 suggestions presented to user.                  System Prompt Strategy: &quot;You are an expert SYSCOHADA accountant for Cameroonian SMEs. Classify the following transaction into the correct SYSCOHADA 2016 category. Return JSON: {category_id, syscohada_code, confidence, category_fr, category_en}. Always respond in the same language as the input.&quot;                  Caching: Identical descriptions cached in Redis with 30-day TTL to avoid redundant API calls. Cache key: SHA256 hash of normalized description.                  Fallback: If AI call fails or times out (&gt;3s), present top 5 most-used categories for that user&apos;s business sector as quick-select options.                    3.1.4 OHADA-Compliant Reports                  Purpose: Auto-generate professional financial documents compliant with SYSCOHADA 2016 standards, suitable for DGI submission and bank presentation.                  Report Types Generated:                      Bilan Comptable (Balance Sheet)          —          Assets, Liabilities, Equity per SYSCOHADA format                      Compte de Résultat (Income Statement / P&amp;L)          —          Revenue, Expenses, Net Income                      Grand Livre (General Ledger)          —          All transactions by account code                      Journal Comptable (Accounting Journal)          —          Chronological transaction register                      Tableau des Flux de Trésorerie (Cash Flow Statement)                      Rapport de Gestion Mensuel (Monthly Management Report with AI commentary)                  PDF Generation: react-pdf library server-side. Template designed with Formalio branding + SYSCOHADA compliance header. All amounts in XAF/FCFA. French primary language with optional English translation.                  Storage: Generated PDFs stored in Supabase Storage (S3-compatible). URL signed with 24-hour expiry for secure sharing. Copy permanently retained for audit trail.                    3.1.5 Tax Filing Assistant                  Purpose: Guide SME owners through Cameroonian tax obligations          —          Impôt Libératoire, Taxe sur la Valeur Ajoutée (TVA 19.25%), Déclaration Statistique et Fiscale (DSF)          —          without professional intervention.                  Workflow: Tax Center screen          →          system analyzes business revenue, sector, and NIU status          →          Claude Haiku determines applicable tax regime          →          generates pre-filled DSF form data          →          provides step-by-step DGI filing instructions          →          alerts user to upcoming deadlines with push notifications.                  Compliance Scope: Impôt Libératoire thresholds (10M FCFA/year); Régime Simplifié (10M         –         50M FCFA); Régime Réel (50M+ FCFA); TVA registration triggers; CNPS contribution reminders for registered employees.                    3.2 Credit Scoring System             —             &quot;Le Score Mosika&quot;                  Purpose: Calculate a proprietary creditworthiness score for each business based on transaction history, enabling access to formal credit from banks and MFIs.                  Score Range: 0         –         1000. Bands: 0         –         400 (Faible / Insufficient), 401         –         600 (Moyen / Moderate), 601         –         800 (Bon / Good), 801         –         1000 (Excellent / Excellent). Integration Approach (Phase 1          —          CSV/SMS Import): User exports MoMo statement as PDF or SMS history          →          uploads to Formalio          →          OCR extracts transactions          →          AI deduplicates against existing records          →          user reviews and confirms          →          bulk import. SECTION 12             —             OFFLINE-FIRST ARCHITECTURE                                12.1 Design Philosophy                  Connectivity in Cameroon is intermittent, expensive, and unreliable          —          especially in semi-urban and rural areas. Formalio&apos;s offline-first design means the app provides full core functionality with zero internet connectivity. Synchronization happens opportunistically when connectivity is available, transparently to the user.                    12.2 PowerSync Architecture                  PowerSync is a synchronization middleware that maintains a local SQLite database on the device and synchronizes bidirectionally with the cloud PostgreSQL database. It handles conflict resolution, ordering guarantees, and incremental sync automatically.                  Data Flow: PostgreSQL (Supabase)          →          PowerSync Sync Service          →          streaming changes          →          Local SQLite (device). Device writes          →          PowerSync upload queue          →          PostgreSQL (Supabase). The device&apos;s SQLite database is the source of truth for reads          —          all UI queries run against SQLite, never blocking on network.                  Tables Synced Offline: transactions (business-scoped), businesses, categories (full dataset), reports (metadata only), notifications (metadata only). Tables Online-Only: audit_logs, credit_scores (computed server-side), subscriptions, ai_logs.                    12.3 Conflict Resolution                  Strategy: Last-Write-Wins (LWW) with server clock as authoritative timestamp. Rational basis: Formalio&apos;s transaction table is append-only by design          —          true conflicts (two clients editing the same record) cannot occur. The only conflict scenario is a client writing while offline, with a server-side update from another source (e.g., admin correction). In such cases, the server version wins and the client is notified.                  Special Cases: If a transaction was &quot;soft-deleted&quot; server-side while a client had it cached, the client receives a deletion sync event and removes it from local SQLite. If the client attempted to edit a record that was server-deleted, the conflict is surfaced as a notification: &quot;Transaction non disponible          —          elle a peut-être été modifiée.&quot;                    12.4 Offline UX Behavior                      Status Indicator: Persistent banner when offline: &quot;Mode hors-ligne actif&quot; with last sync timestamp. Green dot when synced, orange when sync pending, red when sync error.                      Transaction Add: Works fully offline. Shows immediately in local list with &quot;En attente de synchronisation&quot; badge. Badge removed when sync confirmed.                      Report Generation: Queued. Message: &quot;Votre rapport sera généré à la prochaine connexion.&quot;                      Voice Transcription: Audio recorded and saved locally. Transcription deferred. &quot;Transcription en attente&quot; shown.                      AI Categorization: Falls back to user&apos;s most recent 5 manual categories as quick-select. AI result populated when online.                      Credit Score: Cached score displayed with freshness date. &quot;Score calculé le {date}&quot;                    12.5 Sync Queue &amp; Retry Mechanism                  Write Queue: All offline writes stored in SQLite sync_queue table with: operation type (INSERT/UPDATE), table name, payload, created_at, attempt_count. Processing: FIFO order. On connectivity restore, queue processor runs          —          sends operations to API in batches of 20. Exponential backoff: 5s, 30s, 2min, 10min, 1hr (max). After 10 failed attempts, operation moved to sync_errors table for user notification.
